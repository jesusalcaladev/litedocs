# Caching

Boltdocs uses a multi-layered, high-performance caching system to ensure that your documentation builds are as fast as possible, even with thousands of pages.

## Architecture

Our caching system is built on four core pillars:

1.  **Sharded Storage**: Instead of a single massive JSON file, transformation results are stored in individual compressed shards. This drastically reduces I/O overhead.
2.  **Memory-Mapped LRU**: A Least Recently Used (LRU) policy keeps the most active data in memory, preventing leaks in large projects.
3.  **Background Writes**: Cache persistence happens in a non-blocking background queue, so it never slows down your build.
4.  **Batch Reading**: Concurrent loading of shards maximizes the throughput of modern SSDs.

## Configuration

You can tune the caching behavior using the following environment variables:

| Variable | Default | Description |
| :--- | :--- | :--- |
| `BOLTDOCS_NO_CACHE` | `0` | Set to `1` to completely disable the persistent cache. |
| `BOLTDOCS_CACHE_DIR` | `.boltdocs` | The directory where cache files are stored. |
| `BOLTDOCS_CACHE_LRU_LIMIT` | `2000` | The maximum number of transformation results to keep in memory. |
| `BOLTDOCS_CACHE_COMPRESS` | `1` | Set to `0` to disable Gzip compression for cache files. |

### Example `.env`

```bash
# Disable caching for a clean CI build
BOLTDOCS_NO_CACHE=1

# Increase memory limit for very large projects
BOLTDOCS_CACHE_LRU_LIMIT=5000

# Use a custom cache directory
BOLTDOCS_CACHE_DIR=.cache/boltdocs
```

## Maintenance

### Clearing the Cache
To clear the cache, simply delete the `.boltdocs` directory (or your custom `BOLTDOCS_CACHE_DIR`).

```bash
rm -rf .boltdocs
```

### When to Disable Cache
Caching is enabled by default. You might want to disable it if:
- You are debugging MDX transformation issues.
- You want to perform a perfectly "cold" benchmark.
- You are running in an environment with extremely limited disk space.
